# React Error #310 Fix Plan - Edyfra Platform

## Problem Understanding
React error #310: "Render was interrupted by an update from a different component" occurs when:
- A component starts rendering but before it finishes, a state update is triggered from a different part of the tree
- State updates happen synchronously during the render phase instead of inside useEffect or event handlers
- Inconsistent hook definitions (hooks defined after conditional returns) cause state to be re-initialized on re-renders

## Root Cause Analysis

### Issue 1: Tutor Layout (`src/app/tutor/layout.tsx`)
**Problem:** State and useEffect for `isMobileMenuOpen` are defined AFTER the early return statement on line 56.

```typescript
// Lines 21-25: States defined at top
const [loading, setLoading] = useState(true);
const [user, setUser] = useState<User | null>(null);
const [points, setPoints] = useState<number | null>(null);

// Lines 27-29: useEffect defined
useEffect(() => { checkTutor(); }, []);

// Lines 56-60: EARLY RETURN based on loading
if (loading) return <Spinner />;

// Lines 71-83: isMobileMenuOpen state defined AFTER early return with its own useEffect
const [isMobileMenuOpen, setIsMobileMenuOpen] = useState(false);
useEffect(() => { ... }, [isMobileMenuOpen]);
```

**Why this causes error #310:**
- When `loading` transitions from `true` to `false`, the component re-renders
- The second render skips the early return and encounters `isMobileMenuOpen` state definition for the first time in that render cycle
- If the previous `useEffect` (lines 74-83) from the first render is still running/cleanup and tries to update state, or if React's concurrent rendering causes timing issues, a state update can interrupt the render
- More critically: `checkTutor` is NOT wrapped in `useCallback` but is captured in the `useEffect` dependency array indirectly (it's defined before the useEffect but changes every render since it's not memoized)

### Issue 2: Admin Layout (`src/app/admin/layout.tsx`)
**Problem:** Already fixed in commit d8bdceb - `isMobileMenuOpen` state and its useEffect are properly placed at the top with other states.

### Issue 3: Study Room Page (`src/app/study-room/[id]/page.tsx`)
**Problem:** While `fetchSession` and `fetchMessages` are wrapped in `useCallback`, the `useEffect` on line 93 has `[getCurrentUser, fetchSession]` as dependencies. However:

- `getCurrentUser` also depends on `supabase` and is wrapped in `useCallback`
- Between renders, if these callbacks change, the effect re-runs
- The cleanup function removes channels but re-subscription could cause rapid state updates

## Solution

### Fix 1: Move `isMobileMenuOpen` state in Tutor Layout
Move the `isMobileMenuOpen` state definition to the top with other states (lines 21-25), before any useEffect or conditional returns.

**File:** `src/app/tutor/layout.tsx`
**Change:** Move lines 71 and related useEffect (lines 74-83) to after line 25, before line 27.

### Fix 2: Memoize `checkTutor` in Tutor Layout
Wrap `checkTutor` in `useCallback` to prevent it from changing on every render and to ensure the `useEffect` on line 27 doesn't need to re-run unnecessarily.

**File:** `src/app/tutor/layout.tsx`
**Change:** 
- Add `const checkTutor = useCallback(async () => { ... }, [supabase, router]);` 
- Update useEffect dependency: `useEffect(() => { checkTutor(); }, [checkTutor]);`
- Remove the old function definition (lines 31-54) and place it after states

### Fix 3: Optimize Study Room useEffects
Ensure the effects have proper cleanup and don't cause rapid re-subscription loops.

**File:** `src/app/study-room/[id]/page.tsx`
**Change:** Add proper dependency arrays and ensure cleanup functions are comprehensive.
